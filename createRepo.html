Create a Local Repository

Initialize git repository (git init)
  1. Download files from here . In a clean directory named “work”, unjar the contents. You will see it is a simple source tree with a few other things thrown in.
  2. At the command line, in the directory in which you unjarred the given file, execute the below command to initialize repository
    $ git init
  3. Check out the contents of this hidden directory:
    $ ls –la .git

This command creates an empty Git repository - basically a .git directory with subdirectories for objects, refs/heads, refs/tags, and template files. An initial HEAD file that references the HEAD of the master branch is also created.

If you list out the files in .git, you would be able to see the files :COMMIT_EDITMSG, config, description, HEAD, hooks, index, info, logs, objects, packed-refs, refs.

Add contents to git repository (git add)
  1. Execute the commands to add contents to the repository
    $ git add *.java
    $ git add <the rest of the stuff>

This command updates the index using the current content found in the working tree, to prepare the content staged for the next commit. It typically adds the current content of existing paths as a whole, but with some options it can also be used to add content with only part of the changes made to the working tree files applied, or remove paths that do not exist in the working tree anymore.

On executing "init add <filename>" command we have the file ready to be committed.

Check repo status (git status)
  1. Git didn’t tell you anything when you did the add. So, do we have stuff in the repo? Let’s ask git:
    Verify the status of the Git repository with the following command
    $ git status

A whole bunch of stuff comes back with the header:

#On branch master

#Initial commit

#

#Changes to be committed:

#(use "git rm --cached <file>..." to unstage)

and a bunch of “new file: <your file>” lines. What does it mean? Well, git is aware you are adding new files, but says they are not committed. In git terms these files are staged (note the last line about how to unstage a file). So let’s commit.

Commit repo contents (git commit)
  1. Store the changes to Git repository with the following command
    $ git commit –m “Initial commit of my stuff” .

You should get a message back saying something like

[master (root-commit) 9c8baf6] Initial commit of test source 31 files changed, 4782 insertions(+), 0 deletions(-)

create mode …

Followed by a bunch of the create mode lines, one per each file committed. Now do a git status again, what do you see? Do a “du –s –h .git”, what do you get? Do a “du –s –h .” what do you get? Note how the .git directory is smaller than the rest of the working directory yet it has an entire copy of your stuff plus some metadata. Git does some compression when storing objects to save space. (Note: du -s -h shows you the amount of disk space you are using under a directory. You can get this information other ways as well.)

Wrap-up: Check your status now
  1. Check your status again:

    2. $ git status
    and you should see your previously staged files have been committed to your repo

    OK great! You have initialized a new repo, added (staged) some files and directories to it, and then committed those changes to make them permanent (but undoable, as we will see). Now what happens when you need to change a source file? Continue to find out!
