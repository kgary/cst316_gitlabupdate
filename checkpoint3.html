Checkpoint 3

We’ve talked about the following git commands:

  • clone – Clone (copy) a remote repository and init at the same time
  • remote - issue a command on the remote repository to which this local repository refers to
  • push – port local changes to the remote referred to repo
  • pull – port changes from the remote repo to local repo
  • fetch – fetches changes from the remote repo. Pull is actually a fetch followed by a merge. You can also tell pull to optionally use rebase instead of merge

Managing remote repositories is critically important and fundamentally different than traditional SCC platforms. Git, as a DVCM, supports local repos managing local change. Remote repositories are peers, not master copies as in the traditional client-server paradigm. In effect, with the remote commands you are managing your relationships with peer repositories. This is very very important to understand, and often a source of initial confusion when using Git. Unfortunately I don't think Git helps itself, as default branch labels like master are misleading, and convenience operations hide remote references you created earlier.

Together with the branching/merging of checkpoint 2, remote repo management gives Git its power to define flexible collaborative workflows. You decide where and in what repos stable codelines live, or experimental code, or whatever. You decide who gets what access to the codelines. You define hooks to help enforce your quality and reporting policies on change management. When you truly understand Git as a DVCM, you begin to see the power of collaboration it provdies to you, and why developer communities, particularly open (source) communities, flock to it. Unfortunately, I would say a majority of Git users do not see this - they see GitHub as a nice web interface and simply use it in the old centralized branch and merge workflow. I encourage you to read about Git workflows, including but not limited to the GitHub Flow which are useful when used in conjunction with code reviews.
